# Проект "Stealth Acrobat" (AMES Test Solver)

## Главная цель проекта (User Case)

Основная задача проекта — предоставлять студентам незаметный и быстрый способ решения онлайн-тестов (например, в системах AMES / NSUTS) во время экзаменов или контрольных работ в компьютерных классах, минимизируя риск быть замеченными преподавателями.

### Полный User Case
1. **Экзамен:** Студент сидит за компьютером и проходит сложный тест с множественным выбором.
2. **Копирование:** Студент незаметно выделяет весь текст теста на экране (вопросы и варианты ответов) и копирует его в буфер обмена (`Ctrl+C`).
3. **Маскировка:** Студент открывает заранее подготовленную вкладку с приложением **Stealth Acrobat**. Визуально сайт на 100% выглядит как обычная онлайн-читалка PDF-файлов (интерфейс Adobe Acrobat), поэтому проходящий мимо преподаватель не заподозрит неладного.
4. **Загрузка (Dummy UI):** Студент нажимает `Ctrl+V` (вставляет текст) в любом месте открытого сайта. Оригинальный текст теста мгновенно отправляется на сервер для фонового решения ИИ.
   - *Для безопасности:* Сразу после вставки на экране отображается визуально правдоподобный, но "фейковый" (dummy) тест на русском языке. Студент может сделать вид, что внимательно читает этот "легальный" материал.
   - В этот момент в правой боковой панели приложения появляются счетчики: сначала «Анализ текста...», затем количество найденных (Found) и решенных (Solved) вопросов.
5. **Фоновое решение (Hybrid RAG + LLM):** 
   - Сервер разбивает скопированный текст на отдельные вопросы (через Gemini).
   - Gemini при извлечении **убирает все маркеры** вариантов ответа (❑, a), b), и т.д.) для унификации формата.
   - Каждый вопрос ищется в знаниях двумя способами: **Vector Search** (семантический) + **BM25** (ключевые слова). Результаты мержатся через **Reciprocal Rank Fusion (RRF)**.
   - ИИ (Gemini) получает top-20 лучших фрагментов из лекций и решает вопрос.
6. **Отображение результата:** Как только ответы на вопросы готовы, они начинают поступать обратно в браузер в реальном времени (через SSE).
   - Фейковый тест плавно заменяется на реальный (скопированный).
   - Правильные варианты отмечаются "звездочкой" (★).
   - Интерфейс автоматически пагинирует вопросы, чтобы они не вылезали за границы PDF-страниц, добавляя разделители.
7. **Умное кэширование:** Если несколько студентов в аудитории сдают один и тот же вариант, сервер не делает повторных запросов к нейросети. Используется **order-independent хэширование** (MD5 после сортировки строк и удаления маркеров), чтобы вопросы с перемешанными вариантами ответов давали одинаковый хэш. Первый студент "захватывает" задачу, остальные получают ответ из кэша или подписываются через **Redis Pub/Sub**.

## Архитектура проекта

Система состоит из нескольких легковесных и отвязанных друг от друга компонентов:

### 1. Frontend (Клиентская часть)
- **Технологии:** Vanilla HTML / CSS / JavaScript (без тяжелых фреймворков).
- **Особенности:** Доскональная имитация интерфейса Adobe Acrobat. Использование **Server-Sent Events (SSE)** для получения ответов по мере их генерации, без постоянной перезагрузки страницы. Автоматическая пагинация по размеру "листа А4". Поддержка Ctrl+Scroll зума.

### 2. Backend Web Server (Flask + Waitress)
- **Роль:** Прием запросов от пользователей (через HTTP/SSE) и отдача статики.
- **Особенности:** Точка входа `/api/solve_stream`. Использование многопоточного WSGI-сервера `waitress` для поддержания долгих (streaming) соединений.
- **Диспетчеризация:** Когда приходит тест, сервер разбивает его на вопросы и раздает микро-задачи в очередь (брокер сообщений).
- **Pub/Sub:** Подписка на Redis каналы `stealth:solved:<hash>` вместо polling для мгновенного получения результатов.

### 3. Smart Deduplication & Cache (Redis)
- **Роль:** Брокер сообщений для Celery, система блокировок (Mutex), кэш ответов в Redis, и **Pub/Sub** для уведомлений.
- **Order-Independent Hashing:** Каждый вопрос хэшируется после: lowercase → удаление маркеров (a), b), ❑, -, * и номеров вопросов до 3 цифр) → сортировка строк → MD5. Это гарантирует, что перемешанные варианты ответа дают одинаковый хэш.
- **Логика работы:**
  - Если вопрос уже есть в кэше (`stealth:ans:<hash>`, TTL 30 мин), ответ выдается моментально.
  - Если ответа нет, используется атомарная блокировка `SET NX` (`stealth:pending:<hash>`, TTL 5 мин), чтобы только **одна** фоновая задача решала вопрос.
  - После решения воркер делает `PUBLISH stealth:solved:<hash>` — все подписчики мгновенно получают уведомление.

### 4. Background Workers (Celery Thread Pool)
- **Роль:** Обработка тяжелых ИИ-задач без блокировки веб-сервера.
- **Особенности пула:** Использует `--pool=threads` с `--concurrency=10`. ML-модели лениво загружаются только *один* раз на весь процесс демона Celery и шарятся между потоками (Double-checked locking mechanism).
- **Embedding Batching:** Вопросы группируются в батчи для более эффективной векторизации.

### 5. Hybrid RAG Pipeline & LLM Integration
- **Retriever — Vector Search (ChromaDB + SentenceTransformers):** Локальная векторная база знаний. Принимает вопрос, ищет наиболее близкие эмбеддинги (`pplx-embed-context-v1-0.6B`) по материалам лекций курса. Возвращает `top-20` чанков.
- **Retriever — BM25 Keyword Search:** Параллельный поиск по ключевым словам через индекс `rank_bm25` (`bm25_index.pkl`, ~12 MB). Ловит точные совпадения терминов, которые embedding-модель может пропустить.
- **RRF Merge (Reciprocal Rank Fusion):** Объединяет результаты vector и BM25 поиска. Чанки, найденные обоими методами, получают наивысший скор. Финальные `top-20` отправляются в Gemini.
- **Reranker:** Отключён (ONNX mMiniLM). Hybrid search + RRF заменяют его функцию.
- **LLM (Gemini):** Отправка контекста (лекции) + структурированного промпта + вопроса к прокси-API Gemini. Настроен **Circuit Breaker** с round-robin балансировкой между прокси-серверами — при ошибках эндпоинт исключается из ротации на 60с.
- **Chunking:** `CHUNK_SIZE=2000`, `CHUNK_OVERLAP=400`, контекстные эмбеддинги (±1 страница).
