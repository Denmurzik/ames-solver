# Stealth Acrobat — AI Test Solver (Hybrid RAG & Distributed System)

## О проекте
Stealth Acrobat — это система скрытого решения университетских онлайн-тестов в реальном времени. Внешне приложение 1:1 копирует интерфейс Adobe Acrobat (PDF-читалка), маскируя свою настоящую функцию. В фоне — это мощный асинхронный конвейер на базе **LLM (Gemini)**, **Hybrid RAG** и **распределенных очередей**.

Проект разрабатывался с упором на отказоустойчивость, высокую нагрузку (одновременная сдача экзамена всем потоком) и микрооптимизации ИИ-инфраструктуры.

---

## Главные фичи и технические решения

1. **Order-Independent Hashing (Умное кэширование)**
   Тестирующие системы перемешивают варианты ответов. Реализован алгоритм хэширования (MD5 после нормализации и сортировки), чтобы разные пользователи с перетасованными вариантами получали мгновенный ответ из кэша (Redis), минуя повторные запросы к нейросети.

2. **Smart Deduplication & Pub/Sub**
   Если несколько человек одновременно загружают тест с одинаковым вопросом, запрос к ИИ делает только *одна* фоновая задача. Остальные "подписываются" на Redis-канал (`stealth:solved:<hash>`) и мгновенно получают ответ по мере его готовности.

3. **Hybrid RAG (Без галлюцинаций)**
   Использован гибридный поиск (**Hybrid Search**): параллельно работает семантический поиск по эмбеддингам (ChromaDB + SentenceTransformers) и лексический поиск (BM25) для точного совпадения сложных узкоспециализированных терминов. Результаты объединяются алгоритмом **Reciprocal Rank Fusion (RRF)** с приоритизацией источников информации.

4. **Отказоустойчивость LLM (Circuit Breaker)**
   Настроена интеллектуальная балансировка (Round-Robin) между пулом прокси-серверов. Если конкретный сервер падает (HTTP 429/503/Timeout), паттерн Circuit Breaker временно "выдворяет" его из ротации.

5. **Streaming Events (SSE)**
   Результаты работы нейросети и RAG-пайплайна возвращаются клиенту асинхронно, по мере их генерации, через Server-Sent Events. Это обеспечивает плавный UX, критически важный в условиях ограниченного времени на экзамене.

## Стек технологий

*   **ИИ / ML:** Gemini 1.5 Flash API, SentenceTransformers (`pplx-embed-context-v1-0.6B`), ChromaDB (Векторная БД), BM25, Reciprocal Rank Fusion (RRF).
*   **Backend:** Flask, Waitress (многопоточный WSGI), Celery (Background Workers).
*   **Очереди и Кэш:** Redis (Task Broker, Pub/Sub, Atomic Locks, Distributed Cache).
*   **Инфраструктура:** Thread Polling, Батчинг инференса ML-моделей (Batched Embeddings).
*   **Frontend:** Vanilla JS, Server-Sent Events (SSE), полное копирование UI Adobe Acrobat (Dummy UI).

## Архитектура (Под капотом)

### 1. Парсинг и извлечение вопросов 
Пользователь копирует страницу теста и вставляет её (Ctrl+V) на сайт. Фронтенд подменяет текст на "dummy" (фейковый текст лекции для визуальной маскировки), а оригинал отправляется на бэкенд. Нейросеть вычищает системный мусор (маркеры, чекбоксы) и разбивает полотно на массив структурированных JSON-вопросов.

### 2. Дедупликация и Роутинг задач
Celery-бэкенд проверяет Redis-кэш. Если вопроса нет — ставит `SET NX` блокировку на 5 минут в кэше. Свободный воркер берёт задачу в работу. Любые дублирующие запросы с этим вопросом становятся подписчиками (Subscribers) и не напрягают API.

### 3. Инференс ML и Ретривер (Hybrid RAG)
Вопросы собираются в логические батчи для ускорения работы эмбеддингов. Запрашивается векторная база ChromaDB и локализованный индекс BM25. Найденные контексты (чанковые куски лекций) проходят скоринг RRF и добавляются в системный промпт. В памяти ML-модели загружаются "лениво" (Lazy Loading) с использованием локов, чтобы не блокировать оперативную память.

### 4. Генерация и Стриминг (SSE)
На основе собранного контекста LLM анализирует каждый вариант ответа. Как только первый ответ готов, воркер делает `PUBLISH` в Redis. WSGI-сервер (Waitress), слушающий канал, "прокидывает" готовый фрагмент на фронтент через SSE. Правильные варианты плавно появляются в интерфейсе пользователя в реальном времени.
